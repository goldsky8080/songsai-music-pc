# 아키텍처 기준서

## 중심 원칙

- `songsai-api`가 진짜 원본 백엔드다.
- 프론트는 가능한 한 API 호출만 담당한다.
- `D:\music`은 참고용이다.

## 음악 생성 흐름

1. 프론트가 `POST /api/v1/music` 호출
2. 백엔드가 `Music`, `GenerationJob(MUSIC_GENERATION)` 저장
3. worker가 큐를 처리해 provider에 생성 요청
4. 완료/확정 전까지는 `QUEUED` 또는 `PROCESSING`
5. 완료 시점에 `mp3Url`, `timed lyrics`, 필요 시 waveform을 저장

## 홈과 자산 화면 차이

### 홈

- `COMPLETED + mp3Url 있음` 조건의 완료곡만 표시
- 최근 완료곡 슬라이더와 공용 플레이어 제공

### My Assets

- 내가 생성한 곡 중심
- `QUEUED / PROCESSING / COMPLETED / FAILED` 모두 표시
- 불안정 URL이라도 `mp3Url`이 있으면 듣기 허용
- 다운로드는 `downloadAvailableAt` 이후 허용

## 커버 이미지 정책

- 캐시 전: 기존 provider 이미지 URL 사용
- 캐시 후: 우리 서버 URL 사용 가능
- 캐시가 실제로 없는데 우리 서버 URL로 먼저 바꾸지 않는다

## 비디오 정책

- provider `videoUrl` 의존 구조로 가지 않는다
- 입력 자산: mp3, cover image, timed lyrics
- 처리 방식:
  - `VIDEO_RENDER` job 생성
  - worker가 ffmpeg 기반 서버 렌더 수행
  - 완료 시 `Video`와 `MusicAsset` 저장
