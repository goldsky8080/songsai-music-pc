# 아키텍처 기준서

## 역할 분리

- `songsai-api`가 실제 백엔드다.
- `songsai-music-pc`는 `songsai-api`를 호출하는 프론트엔드다.
- `D:\music`은 UI와 생성 흐름을 참고하는 레퍼런스 프로젝트다.

## 음악 생성 기준

1. 프론트가 `POST /api/v1/music`를 호출한다.
2. 백엔드는 요청을 검증한 뒤 Suno provider를 직접 호출한다.
3. 생성 직후 Suno가 내려주는 초기 데이터가 있으면 즉시 `Music`에 저장한다.
4. 이때 저장 대상은 최소 `providerTaskId`, `title`, `mp3Url`, `imageUrl`, `tags`, `duration`이다.
5. `MUSIC_STATUS_POLL` job은 생성 자체를 담당하지 않고, 후속 상태 확인과 메타데이터 갱신만 담당한다.

즉 생성의 기본 축은 내부 큐가 아니라 `provider 직접 호출`이다.

## 429 처리 기준

- Suno가 `429 Too Many Requests`를 반환하면 생성 실패로 바로 응답한다.
- 사용자에게는 `현재 생성 대기열이 가득 차 있습니다. 잠시 후 다시 시도해 주세요.`로 안내한다.
- 이 경우 생성 성공으로 간주하지 않는다.
- 크레딧 정책을 붙일 때도 `429`, validation error, auth error는 미차감 기준으로 본다.

## 생성 후 5분 정책

### 5분 전

- 미리듣기만 허용한다.
- 다운로드와 비디오 생성은 비활성화한다.
- 미리듣기에는 Suno가 생성 시점에 내려준 초기 `mp3Url`을 우선 사용한다.
- 초기 `mp3Url`, `title`, `imageUrl`이 비어 있으면 최근 목록 조회나 inline preview 시점에 provider 상태를 가볍게 다시 확인할 수 있다.
- 이 구간에서는 파일 다운로드, 서버 캐시, 최종 완료 처리를 하지 않는다.

### 5분 후

- 미리듣기, 다운로드, 비디오 생성 진입 시 provider 상태를 다시 확인한다.
- 최신 `title`, `mp3Url`, `imageUrl`, `tags`, `duration`, timed lyrics를 DB에 반영한다.
- 필요한 파일을 우리 서버에 저장한다.
- 이 시점부터 최종 완료 처리와 `MusicAsset` 갱신이 가능하다.

## 다운로드 / inline playback 기준

- `GET /api/v1/music/:id/download?inline=1`
  - 5분 전에는 raw preview URL 연결만 허용한다.
  - 5분 전에는 절대 mp3/image/lyrics 파일을 서버에 저장하지 않는다.
  - 5분 후에는 최신 상태 확인 후 local asset 또는 최신 URL로 재생한다.

- `GET /api/v1/music/:id/download`
  - 5분 전에는 차단한다.
  - 5분 후에는 provider 상태를 다시 확인한 뒤 최신 URL과 메타데이터를 반영한다.
  - 필요 시 mp3 lazy cache를 수행한다.

## 제목 / 메타데이터 기준

- 생성 직후 Suno가 `title`을 주면 그 제목을 바로 사용한다.
- 생성 직후 제목이 없으면 임시 fallback 제목을 사용한다.
- 5분 후 듣기/다운로드/비디오 생성 시점에 provider가 준 최신 `title`로 갱신한다.
- `tags`, `duration`도 같은 시점에 함께 갱신한다.

## MusicAsset 기준

- 앱 코드가 `createdAt`만 보고 hot/archive 경로를 직접 고르지 않는다.
- 자산 위치와 해석은 `MusicAsset` 메타데이터로 관리한다.
- 기준 컬럼은 `storageTier`, `storageKey`, `storagePath`, `publicUrl`, `sourceUrl`, `archivedAt`이다.

## 캐시 기준

- 커버 이미지: 5분 후 최종 갱신 시점에 우리 서버로 캐시
- 타임라인 가사: 5분 후 최종 갱신 시점에 우리 서버로 캐시
- mp3: 다운로드 또는 비디오 생성 시점에 lazy cache

## 스토리지 기준

- SSD 1TB: hot storage
- NAS 4TB: archive storage
- 3개월 기준 이동 정책은 요청 처리 코드가 아니라 별도 아카이브 작업에서만 사용한다.

## 비디오 기준

- provider `videoUrl` 직접 사용 구조를 최종 구조로 보지 않는다.
- 입력 자산은 `mp3 + cover image + timed lyrics` 기준으로 본다.
- 비디오 생성은 `VIDEO_RENDER` job과 ffmpeg 기반 서버 렌더를 최종 방향으로 둔다.

## UI 기준

- 상단 메뉴는 `Home / Create / My Assets / Pricing / Support`
- `Create`는 음악 생성 중심 화면
- `My Assets`는 생성 결과와 진행 상태 관리 화면
