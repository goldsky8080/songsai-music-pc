# Support 운영형 문의 화면 설계

기준일: 2026-04-14

## 목적

`Support` 메뉴를 템플릿성 연락처 페이지가 아니라, 실제 서비스 운영 문의를 접수하는 화면으로 전환한다.

이 화면은 오프라인 방문 안내나 매장 위치 안내보다 아래 목적에 집중한다.

- 계정/로그인 문제 접수
- 결제/크레딧 문의 접수
- 음악 생성 문제 접수
- 비디오 생성 문제 접수
- 다운로드/자산 문제 접수
- 기타 운영 문의 접수

## 방향

### 1. 지도는 1차에서 제외

현재 `Support` 성격은 오프라인 위치 안내보다 운영 지원 채널에 가깝다.

따라서 1차 구현에서는:

- 지도 제거
- 주소/매장 위치 중심 레이아웃 제거
- 운영형 문의 폼 중심 화면으로 구성

### 2. 문의 폼 중심 구조

핵심은 사용자가 문제를 정확히 남기고, 운영자가 바로 확인할 수 있게 만드는 것이다.

권장 구조:

- 상단 Hero
  - 제목: `Support`
  - 설명: `계정, 생성, 다운로드, 비디오 관련 문의를 남겨주세요.`

- 본문 왼쪽
  - 문의 폼

- 본문 오른쪽
  - 접수 안내
  - 자주 묻는 문의 유형
  - 답변 예상 시간

## 1차 화면 구성

### Hero

- 짧은 제목과 안내 문구만 둔다.
- 과한 홍보 문구나 템플릿용 문구는 제거한다.

예시:

- 제목: `Support`
- 보조 문구: `문제 상황을 남겨주시면 운영 흐름에 맞춰 확인합니다.`

### 문의 폼

필드 제안:

1. 문의 유형
   - 계정
   - 결제/크레딧
   - 음악 생성
   - 비디오 생성
   - 다운로드/자산
   - 기타

2. 이름
   - 로그인 사용자는 기본값 자동 채움 가능

3. 회신 이메일
   - 로그인 사용자는 계정 이메일 자동 채움
   - 비로그인 사용자는 직접 입력

4. 제목
   - 짧은 문의 제목

5. 문의 내용
   - 상세 설명

6. 관련 곡 선택 또는 관련 곡 ID
   - 로그인 사용자는 최근 곡 목록 중 선택 가능
   - 1차에서는 선택형 또는 숨김 연결 둘 중 하나

7. 첨부파일
   - 1차에서는 선택
   - 스크린샷, 오류 캡처 첨부 대비

### 오른쪽 안내 영역

예시 내용:

- 어떤 문의를 받을 수 있는지
- 답변은 운영 상황에 따라 순차 처리된다는 점
- 로그인 후 문의하면 계정/곡 흐름과 연결해 더 빠르게 확인 가능하다는 점

## 사용자 상태별 동작

### 로그인 사용자

- 이름/이메일 자동 채움
- 관련 곡 선택 기능 연결 가능
- 제출 시 계정 ID와 함께 저장

### 비로그인 사용자

- 이름/이메일 직접 입력
- 일반 문의로 접수
- 로그인 유도 문구는 약하게만 둔다

## 백엔드 연결 방향

`Support`는 다음 두 갈래 중 하나로 설계할 수 있다.

### A. DB 직접 저장 + 운영 메일 발송

- `POST /api/v1/support/contact`
- 서버가 DB에 문의 저장
- 동시에 운영 메일 발송

장점:

- 서비스 내부에서 문의 추적이 쉬움
- 관리자 화면과 바로 연결 가능

### B. 운영 메일 webhook 기반 수신함 연결

- 프론트 제출
- 서버가 운영 메일 주소로 문의 메일 발송
- 그 메일이 수신함 webhook으로 다시 저장됨

장점:

- 관리자 수신함과 구조 통일

단점:

- 한 번 더 메일을 거치므로 경로가 길어짐

### 권장

1차는 아래 방식이 가장 깔끔하다.

- `POST /api/v1/support/contact`
- DB에 직접 문의 저장
- 필요 시 운영 메일도 함께 발송
- 관리자 수신함 또는 별도 관리자 문의 목록과 연결

즉 메일 수신함과 완전히 같은 테이블을 재사용하기보다, 문의 제출용 엔드포인트를 따로 두는 편이 운영상 더 깔끔하다.

## 관리자 연결 방향

현재 관리자 영역에 이미 다음이 있다.

- `Members`
- `Inbox`

`Support`를 붙인 뒤에는 관리자 쪽에서 두 방향 중 하나로 간다.

### 1. Inbox 재사용

- 문의 제출 내용을 메일 형식으로 저장해 `Inbox`에서 그대로 확인

### 2. Support Tickets 별도 화면

- `Support Tickets`
- 유형, 상태, 작성자, 관련 곡, 첨부를 따로 보기 좋게 정리

### 권장

운영 관점에서는 최종적으로 `Support Tickets`가 더 낫다.
하지만 1차는 빠르게 가기 위해 `Inbox` 또는 단순 문의 저장 리스트로 시작해도 된다.

## 1차 구현 우선순위

1. `Support` 화면을 운영형 문의 폼으로 전환
2. 지도 제거
3. 로그인 사용자 이메일 자동 채움
4. 문의 유형/제목/내용 필수 검증
5. 제출 API 연결
6. 관리자에서 확인 가능한 저장 구조 연결

## 2차 확장 후보

1. 관련 곡 선택
2. 첨부파일 업로드
3. 문의 내역 조회
4. 문의 상태 표시
5. 운영자 답변/메모

## 현재 판단

- `Support`는 지도 페이지보다 운영형 문의 폼이 우선이다.
- 1차에선 지도 없이 간다.
- 관리자 수신함 1차가 이미 있기 때문에, 이후 운영 연결도 자연스럽게 이어질 수 있다.
