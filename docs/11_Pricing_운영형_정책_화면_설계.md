# Pricing 운영형 정책 화면 설계

기준일: 2026-04-14

## 목적

`Pricing` 메뉴를 템플릿성 이벤트 페이지가 아니라 실제 서비스 정책 안내 화면으로 전환한다.

현재 SongsAI Music은 결제와 크레딧 차감 정책이 완전히 고정된 상태가 아니므로, 1차 `Pricing` 화면은 확정 가격표를 강하게 노출하는 페이지보다 아래 목적에 더 집중한다.

- 현재 제공 기능 안내
- 사용 흐름 안내
- 추후 크레딧/요금 정책의 방향 제시
- 사용자가 `Create`, `My Assets`, `Support`로 자연스럽게 이어지도록 유도

## 방향

### 1. 장식형 이벤트 페이지를 중단

현재 `/pricing`은 템플릿의 `event.html` 기반 연결이다.

이 구조는 음악 제작 서비스의 정책/요금 안내와 맞지 않으므로, `Support`처럼 별도 React 페이지로 전환하는 것이 맞다.

권장 구조:

- `app/pricing/page.tsx`
- `app/pricing/pricing-studio.tsx`
- `app/pricing/pricing-studio.module.css`

### 2. "지금 운영 단계"에 맞는 안내형 Pricing

1차 Pricing은 확정되지 않은 결제 상품을 과하게 약속하지 않고, 현재 서비스 흐름을 설명하는 정책형 페이지로 간다.

예를 들어 아래 내용을 중심으로 보여준다.

- 계정 기반 사용
- 음악 생성 후 5분 전/후 동작 차이
- 다운로드/비디오 생성 방식
- 이메일 인증 계정 기준 운영
- 향후 크레딧 기반 확장 예정

즉 1차는 "지금 무엇을 쓸 수 있는지"와 "향후 어떤 구조로 갈지"를 분리해서 보여주는 것이 중요하다.

### 3. 지도/오프라인 정보 없음

`Pricing`은 오프라인 공간, 공연장, 이벤트형 페이지가 아니므로 지도, 장소, 날짜 블록은 제외한다.

### 4. 사용자 행동 유도 중심

페이지를 본 뒤 사용자가 실제로 해야 할 행동은 아래 중 하나다.

- Create로 이동
- My Assets에서 결과 확인
- Support로 문의

따라서 CTA는 이 세 흐름과 연결되는 것이 좋다.

## 1차 화면 구성 제안

### Hero

- 제목: `Pricing`
- 보조 문구:
  - `SongsAI Music의 현재 사용 방식과 향후 크레딧 구조를 안내합니다.`

Hero는 과한 홍보 문구보다 현재 운영 정책을 간단히 설명하는 톤이 맞다.

### 섹션 1. 현재 제공 기능

카드 3개 또는 4개 정도로 구성:

- 음악 생성
- 미리듣기
- 다운로드
- 비디오 생성

각 카드에는 실제 동작을 짧게 설명한다.

예시:

- 음악 생성: `로그인 계정 기준으로 생성 이력이 누적됩니다.`
- 미리듣기: `생성 직후에는 미리듣기 중심으로 먼저 확인할 수 있습니다.`
- 다운로드: `일정 시간이 지난 뒤 완전한 자산 기준으로 내려받을 수 있습니다.`
- 비디오 생성: `음악, 이미지, 가사 자산을 바탕으로 서버에서 비디오를 생성합니다.`

### 섹션 2. 현재 운영 정책

리스트 또는 강조 블록으로 안내:

- 이메일 인증 계정 기준 사용
- 생성 자산은 계정에 연결됨
- 생성 직후와 최종 자산 확정 시점이 다를 수 있음
- 다운로드/비디오 생성은 자산 확정 이후 기준으로 제공

이 섹션은 실제 서비스 흐름을 이해시키는 역할을 한다.

### 섹션 3. 향후 요금/크레딧 방향

1차에서는 확정 가격 대신 정책 방향만 정리한다.

예시:

- 무료 사용 구간 또는 체험 구조 검토 중
- 크레딧 기반 생성 구조로 확장 예정
- 고급 생성/비디오/추가 기능은 이후 분리 가능
- 요금 정책은 추후 `Pricing`에서 공지

중요한 점은 "아직 확정되지 않은 숫자"를 억지로 넣지 않는 것이다.

### 섹션 4. 자주 묻는 질문

FAQ 예시:

- 왜 생성 직후와 다운로드 가능 시점이 다른가요?
- 비디오는 어디에서 만들어지나요?
- 계정 인증이 필요한 이유는 무엇인가요?
- 문의는 어디로 남기나요?

### 섹션 5. 하단 CTA

권장 CTA:

- `Create 시작하기`
- `My Assets 보기`
- `Support 문의하기`

## 화면 톤 제안

`Support`처럼 운영형 페이지이지만, 너무 건조한 문서형 UI보다는 현재 홈/헤더 톤과 자연스럽게 이어지는 모던한 섹션형 페이지가 좋다.

권장 톤:

- 어두운 히어로 + 밝은 본문 섹션 조합
- 과한 이벤트/콘서트 느낌은 배제
- 텍스트 위주지만 카드와 배지로 리듬감 부여
- 보라/핑크 계열 기존 포인트 컬러는 CTA에만 제한적으로 사용

## 백엔드 연결 방향

1차 Pricing은 읽기 중심 페이지이므로 별도 결제 API 연결은 필요 없다.

다만 이후를 고려하면 아래 정보와 연결될 가능성이 있다.

- 현재 사용자 role
- 현재 크레딧/플랜
- 사용량
- 결제 이력

따라서 1차 구현에서는 UI를 정적으로 두되, 추후 아래 API가 붙을 수 있도록 섹션을 분리해두는 것이 좋다.

- `GET /api/v1/billing/summary`
- `GET /api/v1/billing/history`
- `POST /api/v1/billing/checkout`

## 1차 구현 우선순위

1. `/pricing`를 별도 React 페이지로 전환
2. Hero + 현재 제공 기능 섹션 구현
3. 현재 운영 정책 섹션 구현
4. 향후 요금/크레딧 방향 섹션 구현
5. FAQ + CTA 구성

## 2차 확장 후보

1. 실제 크레딧 잔액/사용량 연결
2. 요금제 카드 추가
3. 충전/구독 버튼 추가
4. 결제 내역/사용 내역 화면 연결
5. 관리자용 가격 정책 편집 흐름

## 현재 결론

- `Pricing`은 단순 가격표보다 운영형 정책/안내 페이지로 먼저 가는 것이 맞다.
- 1차에서는 숫자보다 사용 방식과 정책 흐름을 명확히 보여주는 것이 중요하다.
- 이후 결제/크레딧이 구체화되면 같은 페이지에 실제 요금제 섹션을 확장하면 된다.
