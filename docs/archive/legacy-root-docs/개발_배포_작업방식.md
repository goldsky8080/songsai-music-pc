# 개발·배포 작업방식

이 문서는 앞으로의 기본 개발 방식과 서버 반영 방식을 정리합니다.

## 기본 원칙

메인 개발 환경은 로컬입니다.

현재 로컬 작업 기준:

- `D:\music`
- `D:\wrapper\suno-api`
- `D:\songsai-bank-bridge`

신규 서버는 주 개발 환경이 아니라, 반영과 검증 대상입니다.

## 기본 흐름

기본 작업 순서는 아래와 같습니다.

1. 로컬에서 수정
2. 로컬에서 테스트
3. git commit
4. GitHub push
5. 신규 서버에서 `git pull`
6. migration / 재시작 / 확인

## 왜 이 방식이 좋은가

- 로컬이 더 빠르고 안전하다
- git 이력이 남는다
- 서버가 덜 꼬인다
- 롤백과 비교가 쉽다
- 서버 직접 수정이 줄어든다

## 각 환경의 역할

### 로컬 PC

- 기능 개발
- UI 수정
- 테스트
- 커밋과 푸시

### 신규 서버

- pull
- 설치
- migration
- 실행 확인
- 다음 배포 대상

### N100 서버

- 현재 legacy 운영 기준
- 기존 `.env`와 운영 동작 참고 기준

## 권장 배포 방식

기본은 git 배포입니다.

즉:

- 로컬 수정
- git push
- 서버 pull

## FTP 사용 규칙

FTP나 직접 파일 전송은 예외 상황에서만 사용합니다.

예:

- 긴급 단일 파일 패치
- 비코드 파일 전달
- 임시 산출물 복사

기본 배포 수단으로 FTP를 쓰지는 않습니다.

## 서버 직접 수정 규칙

소스코드는 서버에서 직접 고치지 않는 것을 원칙으로 합니다.

서버에서 직접 수정해도 되는 영역:

- `.env`
- Nginx 설정
- 서비스 실행 설정
- 꼭 필요한 운영성 임시 조치

## git 작업 규칙

- 큰 작업은 브랜치 사용 권장
- 관련 없는 변경은 한 커밋에 섞지 않기
- 생성 산출물은 일반 코드 커밋에 넣지 않기

## 산출물 관리 규칙

아래 같은 파일은 일반 소스와 분리해서 관리합니다.

- `public/generated-*`
- `storage/`
- 대용량 미디어 출력물

이런 파일은 서버 저장소나 아카이브 영역으로 관리하는 것이 맞습니다.

## `.env` 규칙

- `.env`는 서버별로 다를 수 있다
- 실제 비밀값은 git에 넣지 않는다
- 기존 서버 값을 복사해 올 때는 아래를 꼭 다시 확인한다
  - URL
  - 포트
  - DB 이름
  - 저장 경로

## 현재 기준 결론

앞으로도 기본 워크플로는 아래처럼 유지합니다.

- 로컬에서 수정
- git으로 버전 관리
- 신규 서버에 pull 반영

새 구조로 이동하더라도 이 작업방식은 그대로 유지하는 것이 좋습니다.
