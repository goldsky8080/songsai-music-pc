# 저장소 전략 및 아카이브 정책

## 목표

- `songsai-api`가 생성한 음악/이미지/비디오 자산을 안정적으로 저장한다.
- 최근 자산은 SSD에서 빠르게 서비스한다.
- 오래된 자산은 NAS로 옮겨 장기 보관한다.
- 앱 코드는 물리 저장 위치를 직접 알지 않게 한다.
- 가능하면 파일 이동 시 DB의 공개 URL은 바꾸지 않는다.

## 물리 저장소 역할

### SSD 1TB

용도:

- 최근 생성된 음악 자산
- 최근 생성된 커버 이미지
- 최근 생성된 비디오
- 비디오 렌더링 임시 파일
- 다운로드 직후 자주 접근하는 hot asset

경로:

- `/srv/media/music/...`
- `/srv/media/image/...`
- `/srv/media/video/...`
- `/srv/media/tmp/...`

### NAS 4TB

용도:

- 3개월 이상 지난 장기 보관 자산
- 오래된 음악/이미지/비디오
- 보존용 원본

경로:

- `/srv/oldmedia/music/...`
- `/srv/oldmedia/image/...`
- `/srv/oldmedia/video/...`
- `/srv/oldmedia/tmp/...`

### HDD 500GB

현재는 주 저장소로 쓰지 않는다.

권장 용도:

- 임시 백업
- 수동 백업
- 로그/실험 파일

## 중요한 원칙

앱 코드에서 `createdAt`만 보고 `/srv/media` 또는 `/srv/oldmedia`를 직접 선택하지 않는다.

이유:

- 실제 파일 이동 시점과 생성 시점이 항상 일치하지 않는다.
- 일부 파일은 예외적으로 더 오래 SSD에 남을 수 있다.
- 일부 파일은 더 빨리 NAS로 옮겨질 수 있다.
- 코드 여러 곳에 기간 기준 분기가 퍼지면 관리가 어려워진다.

따라서 서비스 코드는:

- `storageKey`
- `storageTier`
- `publicUrl`

이 세 값을 기준으로 파일을 다룬다.

## 권장 DB 설계

대상:

- `MusicAsset`
- 나중에 필요하면 `Video`

### MusicAsset 권장 컬럼

- `id`
- `musicId`
- `assetType`
- `status`
- `storageKey`
- `storageTier`
- `storagePath`
- `publicUrl`
- `mimeType`
- `fileSize`
- `checksum`
- `sourceUrl`
- `createdAt`
- `updatedAt`
- `archivedAt`

### 필드 의미

#### storageKey

앱이 자산을 식별하는 논리 키.

예시:

- `music/2026/04/cmnuk191z/audio.mp3`
- `image/2026/04/cmnuk191z/cover.jpg`
- `video/2026/04/cmnuk191z/render.mp4`

#### storageTier

자산이 현재 어느 물리 저장 계층에 있는지 표시.

권장 값:

- `hot`
- `archive`

#### storagePath

서버 내부 실제 파일 경로.

예시:

- `/srv/media/music/2026/04/cmnuk191z/audio.mp3`
- `/srv/oldmedia/music/2026/04/cmnuk191z/audio.mp3`

#### publicUrl

프론트 또는 다운로드 API가 사용 가능한 공개 URL 또는 표준 접근 경로.

예시:

- `/media/music/2026/04/cmnuk191z/audio.mp3`
- `/media/image/2026/04/cmnuk191z/cover.jpg`
- `/media/video/2026/04/cmnuk191z/render.mp4`

#### sourceUrl

최초 공급자 원본 URL.

예시:

- Suno mp3 URL
- Suno image URL

#### archivedAt

NAS 등 장기 저장소로 이동한 시각.

## publicUrl 설계

가장 좋은 방식은 `publicUrl`이 항상 같은 패턴을 유지하는 것이다.

권장:

- `/media/music/...`
- `/media/image/...`
- `/media/video/...`

즉 사용자는:

- SSD에 있든
- NAS에 있든

항상 같은 URL 체계를 보게 한다.

## 실제 파일 해석 방식

서비스 로직은 다음처럼 동작한다.

1. DB에서 `MusicAsset` 조회
2. `storageTier` 확인
3. `hot`이면 `/srv/media/${storageKey}`
4. `archive`면 `/srv/oldmedia/${storageKey}`
5. 해당 파일을 스트리밍 또는 내부 URL로 제공

즉 코드 분기 기준은 `createdAt`이 아니라 `storageTier`다.

## 아카이브 정책

### 기준

- 생성 후 3개월 이상
- `storageTier = hot`
- 최근 접근 이력이 낮은 자산 우선

### 이동 대상

- 커버 이미지
- mp3
- 렌더된 비디오

### 제외 대상

- 진행 중 렌더링 임시 파일
- 최근 3개월 이내 자산
- 아직 실패/재시도 중인 작업 자산

## 아카이브 작업 흐름

별도 배치 또는 관리 작업으로 수행:

1. `createdAt < now - 3개월`이고 `storageTier = hot`인 자산 조회
2. SSD 경로의 실제 파일 존재 확인
3. NAS 대상 경로 생성
4. 파일 이동 또는 복사 후 원본 삭제
5. DB 업데이트

DB 업데이트 항목:

- `storageTier = archive`
- `storagePath = /srv/oldmedia/...`
- `archivedAt = now`

`storageKey`, `publicUrl`은 유지하는 것을 권장한다.

## DB를 안 고치고 해결할 수 있는가

부분적으로 가능하다.

### 방법 1. symlink 유지

- 오래된 파일을 NAS로 옮긴 뒤
- 원래 SSD 경로 자리에 symlink를 둔다

장점:

- DB 수정 최소화

단점:

- 운영자가 파일 시스템을 더 신경써야 함
- 링크 무결성 관리 필요

### 방법 2. 마운트/alias 기반

- `/media/...`는 고정 URL
- nginx 또는 앱이 실제 저장 계층을 해석

장점:

- 사용자 URL 고정
- DB 공개 URL 유지 가능

단점:

- 내부 해석 로직 필요

### 추천

DB는 최소한 `storageTier` 정도는 업데이트하는 편이 좋다.

이유:

- 실제 위치를 DB가 아는 편이 운영상 안전하다
- 장기적으로 정리/복구/마이그레이션이 쉬워진다

단, `publicUrl`은 유지해서 프론트 영향은 없게 한다.

## 음악/이미지/비디오 저장 전략

### 커버 이미지

완료 시점에 바로 우리 서버로 캐시하는 방향 권장.

이유:

- 용량이 작다
- 홈/자산/상세에서 계속 노출된다
- Suno URL이 그대로 보이지 않게 할 수 있다
- 비디오 렌더에도 즉시 재사용 가능하다

### mp3

초기에는 lazy 저장 가능.

즉:

- 다운로드 시점
- 비디오 렌더 시점

에 필요하면 서버 캐시

### 비디오

우리 서버 렌더 결과물이므로 기본적으로 우리 저장소에 저장.

## My Assets / Home 노출 기준

### Home

- `COMPLETED`
- 대표 이미지 존재
- 재생 가능한 URL 존재

### My Assets

- `QUEUED`
- `PROCESSING`
- `COMPLETED`
- `FAILED`

모두 표시 가능

## 2트랙 구조 반영 시 저장 관점

음악 1회 생성 시 보통 2트랙이 오므로, 자산 저장도 트랙 단위로 본다.

예:

- 하나의 생성 세션
- 그 안의 track1, track2

각 트랙별로:

- cover image
- mp3
- timed lyrics
- video

를 개별 자산으로 저장하는 것이 자연스럽다.

## 권장 구현 순서

1. `MusicAsset`에 `storageKey`, `storageTier`, `storagePath`, `publicUrl`, `sourceUrl`, `archivedAt` 확장
2. 커버 이미지 완료 시 서버 캐시 + `Music.imageUrl`을 우리 URL로 변경
3. mp3는 다운로드/비디오 렌더 시점에 lazy 저장
4. 비디오는 항상 서버 렌더 결과를 저장
5. 3개월 아카이브 작업 추가
6. `storageTier` 기반 해석 로직 추가

## 현재 시점 추천 결론

- 메인 저장소: SSD `/srv/media`
- 장기 보관: NAS `/srv/oldmedia`
- 앱은 논리 경로와 DB 메타데이터만 사용
- 코드에서 `createdAt`으로 직접 경로 분기하지 않음
- `createdAt 3개월` 기준은 아카이브 작업에서만 사용
- `publicUrl`은 유지하고 `storageTier`만 바꾸는 구조가 가장 안전함
