# 재구축 방향

이 문서는 SONGSAI를 앞으로 어떤 구조로 발전시킬지 정하는 방향성 문서입니다.

## 핵심 결론

완전히 새로 갈아엎기보다, 새 구조를 만들고 검증된 기능만 이식하는 방식이 가장 좋습니다.

즉:

- 구조는 새롭게 정리
- 검증된 기능은 재사용
- UI는 새롭게 설계
- 운영 중인 legacy는 한동안 참고 기준으로 유지

## 목표 구조

### `songsai-music`

역할:

- 사용자용 프론트
- 모바일 우선 흐름
- API 호출과 결과 표시 중심

장기적으로 줄여야 할 것:

- 비즈니스 로직
- 결제 완료 처리
- 생성 오케스트레이션
- 다운로드 링크 생성 로직

### `songsai-api`

역할:

- 실제 메인 백엔드
- 크레딧, 생성 요청, 결과 관리, 자산, 비디오, 다운로드, 관리자 API 담당
- PayPal 결제 처리 담당
- provider 연동과 wrapper 계약의 중심

현재 판단:

- 앞으로의 핵심 시스템 본체는 `songsai-api`가 되어야 한다.

### 별도 PC 프로젝트

권장 방향:

- PC 버전은 별도 프론트 프로젝트로 분리
- `songsai-api`를 공용으로 사용
- 더 다이내믹하고 프리미엄한 데스크톱 경험 제공

관련 문서:

- [PC 버전 프로젝트 구상](D:\music\docs\PC_버전_프로젝트_구상.md)

### `songsai-bank-bridge`

역할:

- 무통장/SMS 입금 자동화 전용 보조 서비스

범위:

- SMS webhook
- 입금 파싱
- 매칭 후보 계산
- 모니터링

## 제품 방향

### 모바일

- 빠르고 단순하게
- 잔액 확인
- 곡 생성
- 최근 곡 확인
- 다운로드
- 충전 진입

### PC

- 더 다이내믹하고 화려하게
- 스튜디오 같은 느낌
- 자산 패널, 진행 상태, 결과 패널을 풍부하게
- 단순 반응형 축소판이 아니라 별도 경험으로 설계

## 결제 방향

- 기본 결제: PayPal
- 보조 결제: bank transfer + bank-bridge

이유:

- 운영 부담을 줄일 수 있음
- bank-bridge는 백업 경로와 특정 사용자 흐름용으로 유용함

## 기술 방향

당장은 기존 스택을 유지하는 것이 맞다.

- TypeScript
- Node.js
- PostgreSQL
- Prisma
- Docker

지금은 언어나 런타임 교체보다 구조 정리가 우선이다.

## 이전 전략

1. 현재 서버와 로컬 개발 흐름을 안정화한다.
2. 새 기능부터 `songsai-api` 중심으로 넣는다.
3. 기존 `music`의 로직을 하나씩 API화한다.
4. `songsai-music`을 점점 얇은 프론트로 정리한다.

우선순위:

1. 결제와 크레딧 API
2. 생성 요청과 상태 API
3. 다운로드와 자산 API
4. 비디오 API
5. 관리자 API

## 하지 말아야 할 것

- 한 번에 전체를 다시 만드는 것
- 아직 돌아가는 구조를 다 버리는 것
- UI와 백엔드를 동시에 크게 뒤엎는 것

## 요약

가장 좋은 방향은:

- `songsai-api`를 중심 백엔드로 키우고
- `songsai-music`은 모바일 우선의 얇은 프론트로 만들고
- PC는 별도로 더 고급스럽게 확장하고
- `songsai-bank-bridge`는 전용 보조 서비스로 유지하는 것이다.
