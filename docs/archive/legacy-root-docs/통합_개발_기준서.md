# SongsAI 통합 개발 기준서

## 목적

이 문서는 앞으로 `songsai-music-pc`, `songsai-api`, `D:\music` 프로젝트를 함께 참고하면서
구조를 `songsai-api` 중심으로 통합해 나갈 때 기준점으로 사용하는 내부 작업 문서다.

사용자에게 보여주기 위한 소개 문서라기보다, 이후 구현 시 모델/책임/배포/환경 분리 기준이
흩어지지 않도록 유지하는 "작업 기준서"에 가깝다.

---

## 현재 관찰한 저장소 역할

### 1. `D:\songsai-music-pc`

- PC 전용 프론트엔드
- 현재는 `one-music` 템플릿을 `songsai-music` 브랜드로 재구성한 상태
- 로그인, Google 로그인, 운영 배포(`pc.songsai.org`)까지 연결 완료
- 장기적으로는 비즈니스 로직을 두껍게 갖지 않고 `songsai-api`를 호출하는 얇은 프론트가 되어야 함

### 2. `D:\wrapper\suno-api`

- 현재 실질적인 `songsai-api` 작업 저장소
- 인증, 세션, Google 로그인, 기본 CORS, wrapper 호출 기능은 이미 존재
- 하지만 음악 도메인 자체는 아직 raw wrapper 레벨에 가까움
- 현재 Prisma에는 `User`만 있고, 음악/비디오/큐 관련 테이블은 없음

### 3. `D:\music`

- 기존 모바일/앱 중심 작업이 많이 들어간 프로젝트
- 단순 프론트가 아니라, 음악 생성 도메인 모델/Provider 추상화/상태 동기화/크레딧/큐까지 포함한
  더 성숙한 "음악 서비스 레이어"가 존재
- 앞으로 `songsai-api`가 흡수해야 할 핵심 설계 자산은 이 프로젝트에 많이 들어 있음

---

## 핵심 방향

### 결론

앞으로는 아래 구조를 목표로 한다.

- `songsai-api`
  - 인증
  - 음악 생성
  - 음악 상태 조회
  - 최근 생성곡 조회
  - 자산/비디오/큐 상태 관리
  - 프론트가 쓰는 최종 응답 모델 제공

- `songsai-music-pc`
  - 화면/UI 렌더링
  - 사용자 액션 입력
  - `songsai-api` 호출
  - API 응답을 표현하는 역할만 담당

즉, 프론트가 DB를 직접 보거나 raw wrapper 응답을 직접 해석하지 않도록 한다.

## 왜 `D:\music`을 같이 봐야 하는가

현재 `D:\wrapper\suno-api`의 wrapper 계층은 Suno/SONGS raw 응답을 거의 그대로 다루고 있다.

대표 필드:

- `id`
- `title`
- `image_url`
- `lyric`
- `audio_url`
- `video_url`
- `created_at`
- `status`
- `model_name`
- `gpt_description_prompt`
- `prompt`
- `tags`
- `duration`
- `error_message`

이 구조는 wrapper 레벨로는 유용하지만, 프론트가 직접 쓰기에는 불안정하다.

반면 `D:\music`에는 다음과 같은 더 상위 도메인 계층이 존재한다.

### `D:\music\server\music\provider.ts`

- provider raw 응답을 정규화함
- 표준화된 필드 예:
  - `providerTaskId`
  - `status`
  - `mp3Url`
  - `videoUrl`
  - `imageUrl`
  - `imageLargeUrl`
  - `generatedLyrics`
  - `providerPrompt`
  - `providerDescriptionPrompt`

### `D:\music\app\api\music\route.ts`

- DB + provider 결과를 프론트 친화적인 응답으로 재구성함
- 트랙을 `requestGroupId` 단위로 묶음
- 표시 제목 계산
- 실패 항목 필터링
- 보너스 트랙/다운로드 가능 시점/상태 요약 등을 제공

### `D:\music\server\music\service.ts`

- `syncMusicStatuses()`를 통해 진행중인 작업의 상태를 동기화함

### `D:\music\prisma\schema.prisma`

이미 다음 모델들이 설계되어 있음:

- `Music`
- `MusicAsset`
- `Video`
- `GenerationJob`

그리고 상태 enum도 이미 있음:

- `MusicStatus`
- `VideoStatus`
- `QueueStatus`
- `JobType`

이건 `songsai-api`로 가져가기에 매우 좋은 자산이다.

## 현재 `songsai-api`와의 차이

### 현재 `songsai-api` (`D:\wrapper\suno-api`)

- 인증은 있음
- wrapper 호출은 있음
- 사용자 세션도 있음
- 하지만 음악 도메인 저장 구조는 아직 없음
- 최근 생성곡을 제공하려면 현재 기준으로는 raw feed(`/api/get`)를 읽어 가공해야 함

### 목표 `songsai-api`

- raw wrapper API를 그대로 외부에 노출하지 않음
- 내부에서는 wrapper를 쓰더라도 외부에는 도메인 응답을 제공
- `D:\music`의 music domain 레이어를 점진적으로 옮겨옴
- 결국 `songsai-music-pc`는 `songsai-api`의 도메인 응답만 소비

## 최근 생성곡(홈 슬라이더) 방향

### 프론트 요구사항

- 홈 메인 패널에서 최근 생성된 곡을 슬라이드로 보여줌
- 최신순
- 최대 10곡
- 생성 완료된 곡만
- 불안정한 URL이나 생성중인 항목 제외
- 제목, 이미지, 오디오 URL 사용

### 중요한 설계 원칙

프론트가 DB를 직접 보거나 wrapper raw feed를 직접 붙으면 안 된다.

권장 구조:

- 프론트:
  - `GET /api/v1/music/recent?limit=10`
- 백엔드:
  - 내부적으로 DB 기반 또는 wrapper 기반으로 조회
  - 완료 조건 필터링
  - 프론트용 응답으로 재구성

### 현재 시점의 현실적인 중간 단계

`songsai-api`에 아직 `Music` 테이블이 없으므로,
1차 구현은 wrapper feed 기반으로 시작할 수 있다.

조건:

- `status`가 완료 상태여야 함
- `audio_url`이 있어야 함
- `error_message`가 없어야 함
- 최신순 정렬
- 10개 제한

다만 이건 임시 단계다.

### 장기 목표

장기적으로는 raw feed 조회가 아니라 `D:\music` 스타일의 저장 모델을 `songsai-api`에 도입하고,
`Music`/`GenerationJob` 기준으로 최근 생성곡을 제공해야 한다.

즉 홈 슬라이더는 최종적으로 "wrapper 응답"이 아니라 "도메인 읽기 모델"을 소비해야 한다.

## 음악 생성 큐 검토

### 결론

큐는 필요하다. 오히려 지금 시점에는 도입하는 것이 맞다.

### 이유

- 음악 생성 요청은 외부 provider 호출(Suno/SONGS)과 상태 동기화가 섞여 있음
- 사용자가 동시에 여러 요청을 보낼 수 있음
- 생성/재시도/오류/동기화/추가 자산 생성이 앞으로 늘어날 가능성이 큼
- 비디오/커버/정렬 가사 등 파생 작업도 생길 수 있음

### 좋은 점

- 생성 요청과 실제 provider 호출을 분리할 수 있음
- 사용자 응답 속도가 좋아짐
- 재시도 전략을 둘 수 있음
- 비디오 생성, 자산 후처리 등도 같은 작업 테이블 위에 얹을 수 있음
- 실패/재처리/감사 로그 관리가 쉬워짐

### 이미 `D:\music`에 있는 단서

`D:\music\prisma\schema.prisma`에는 이미:

- `GenerationJob`
- `queueStatus`
- `attemptCount`
- `payload`
- `result`
- `errorMessage`

가 존재한다.

`JobType`도 분리되어 있다:

- `MUSIC_GENERATION`
- `VIDEO_RENDER`

즉 큐 설계는 새로 발명하기보다 `D:\music`의 작업 모델을 `songsai-api`로 흡수하는 것이 맞다.

### 추천 큐 방향

#### 1차

- `songsai-api`에 `Music`, `GenerationJob` 도입
- 음악 생성 요청이 오면:
  - `Music` row 생성
  - `GenerationJob(jobType=MUSIC_GENERATION, queueStatus=QUEUED)` 생성
  - API는 즉시 접수 응답

#### 2차

- 별도 worker 또는 poll 기반 job processor가 `QUEUED`를 가져가 provider 호출
- 결과를 `Music`/`GenerationJob.result`에 반영

#### 3차

- `syncMusicStatuses()`와 유사한 상태 동기화 루프 도입
- 완료 후 자산 생성/비디오 렌더/자막 처리로 확장

### 추천 판단

큐는 "나중에 있어도 좋은 기능"이 아니라,
`songsai-api`를 진짜 서비스 레이어로 만들기 위한 핵심 기반으로 보는 게 맞다.

## 추천 통합 순서

### 1단계: 응답 모델 통합

`D:\music\server\music\types.ts`, `provider.ts` 기준으로
`songsai-api`의 공식 음악 응답 스키마를 먼저 정한다.

### 2단계: 최근 생성곡 읽기 API

예시:

- `GET /api/v1/music/recent?limit=10`

초기에는 wrapper 기반이어도 되지만, 응답은 프론트용 도메인 모델이어야 한다.

### 3단계: 음악 생성 도메인 모델 이관

`D:\music\prisma\schema.prisma`의 다음 모델을 `songsai-api`에 옮겨온다.

- `Music`
- `MusicAsset`
- `Video`
- `GenerationJob`

### 4단계: 생성 요청 큐화

- 생성 요청은 queue에 적재
- worker 또는 background processing으로 provider 호출

### 5단계: 프론트 완전 API화

`songsai-music-pc`는:

- 로그인 상태
- 최근 생성곡
- 생성 요청
- 상태 조회
- 다운로드/자산 보기

전부 `songsai-api`만 호출하게 만든다.

## 환경/개발 방식 기준

### 개발

- 프론트: `http://localhost:3000`
- API: `http://localhost:3100`

### 운영

- 프론트: `https://pc.songsai.org`
- API: `https://api.songsai.org`

### 원칙

- 코드 공통
- 환경값만 `.env`로 분리
- 프론트는 API URL만 환경에 따라 달라짐
- 운영 확인은 배포 후
- 일상 개발은 로컬 API + 로컬 프론트에서 진행

---

## 현재 구현/운영 상태 메모

- `api.songsai.org` 배포 및 HTTPS 정상
- `pc.songsai.org` 배포 및 HTTPS 정상
- Google 로그인 동작
- 운영 프론트에서 헤더에 사용자 이름 표시까지 확인
- 로컬 개발은 cross-site cookie 이슈가 있으므로,
  인증 관련 기능은 가급적 `localhost API + localhost 프론트` 조합으로 검증하는 것이 좋음

---

## 이후 구현 시 기억할 것

1. 프론트는 raw wrapper 응답을 직접 쓰지 않는다.
2. `songsai-api`가 공식 도메인 응답을 만든다.
3. `D:\music`의 music domain 설계를 적극적으로 재사용한다.
4. 최근 생성곡은 완료된 항목만 API에서 필터해서 내려준다.
5. 큐는 도입하는 쪽이 맞고, `GenerationJob` 구조를 그대로 활용하는 것이 좋다.
6. `songsai-music-pc`는 화면과 사용자 경험에 집중하고, 상태/저장/큐/동기화 책임은 `songsai-api`가 가진다.
