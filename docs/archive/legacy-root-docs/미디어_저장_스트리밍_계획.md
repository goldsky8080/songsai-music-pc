# 미디어 저장·스트리밍 계획

이 문서는 신규 서버, HDD, NAS를 함께 고려한 저장 및 스트리밍 방향을 정리합니다.

## 기본 원칙

- SSD: 현재 서비스와 빠른 작업용
- HDD: 서버 내부 아카이브용
- NAS: 장기 백업과 저속 정적 저장용

## 권장 배치

### SSD

여기에 두는 것:

- OS
- Docker
- PostgreSQL
- 현재 서비스 중인 앱 코드
- 최근 생성된 미디어
- 임시 렌더 파일
- wrapper 작업 파일

### HDD `/srv/archive`

여기에 두는 것:

- 오래된 mp3
- 오래된 mp4
- 이미지 아카이브
- DB dump
- 로그와 백업

### NAS

여기에 두는 것:

- 장기 백업
- 오래된 결과물 보관
- 코드/문서 스냅샷

## 스트리밍 방향

작은 mp3/mp4 파일 기준으로는 NAS도 정적 파일 저장과 다운로드 용도로 충분히 활용할 수 있습니다.

다만 권장 방식은 다음과 같습니다.

- 앱과 실제 서비스 로직은 신규 서버에서 실행
- NAS는 파일 저장소 또는 백업 저장소로 사용
- 스트리밍 서버를 별도로 둘 경우에도 실행은 신규 서버에서 하고, NAS는 라이브러리 저장소로 연결

## Jellyfin / Plex 판단

현재 우선순위는 다음과 같습니다.

1. 메인 앱 + 정적 파일 서빙
2. 필요하면 Jellyfin 추가
3. Plex는 꼭 필요한 경우만 검토

이유:

- 지금 가장 먼저 필요한 것은 안정적인 다운로드/재생 흐름
- 라이브러리형 스트리밍은 그 다음 단계

## 현재 추천 운영 구조

- 메인 앱: 신규 서버
- wrapper: 신규 서버
- bank-bridge: 신규 서버 또는 별도 운영
- 최근 미디어: SSD
- 오래된 미디어: HDD
- 장기 백업: NAS

## 결론

지금 단계에서는 “정적 파일을 안정적으로 다루는 구조”가 우선입니다.

즉:

- SSD는 현재 서비스
- HDD는 아카이브
- NAS는 장기 저장

이 구성을 기준으로 운영하는 것이 가장 현실적입니다.
