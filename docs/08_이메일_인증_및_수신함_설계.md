# 이메일 인증 및 수신함 설계

## 목적

- 일반 회원가입 시 이메일 인증을 거쳐서만 계정을 활성화한다.
- 발신 주소를 `songsai.org` 도메인으로 통일한다.
- 수신 메일을 우리 서버 DB에 저장하고 관리자 화면에서 조회한다.
- 인증 메일, 운영 알림, 문의 메일을 모두 `songsai-api` 중심으로 관리한다.

## 기준 결론

- 도메인/DNS 관리: Cloudflare
- 메일 발송/수신: Resend
- 앱 백엔드: `songsai-api`
- 발송 주소 예시:
  - 인증 메일: `auth@songsai.org`
  - 자동 발송: `no-reply@songsai.org`
  - 문의 수신: `support@songsai.org`

이 조합을 추천하는 이유:

- Cloudflare는 DNS 관리가 쉽다.
- Resend는 발송과 수신 webhook을 함께 붙이기 쉽다.
- `songsai-api`에서 인증 메일, 재발송, 수신함 저장, 관리자 조회를 한 번에 관리하기 좋다.

## 회원가입 인증 흐름

### 권장 방식

- 회원가입 폼 제출 시 `User`를 바로 만들되 `emailVerifiedAt`는 비운다.
- 동시에 이메일 인증 토큰을 발급하고 인증 메일을 보낸다.
- 사용자가 이메일 링크를 클릭하면 토큰을 검증하고 `emailVerifiedAt`를 채운다.
- 로그인은 `emailVerifiedAt IS NOT NULL`인 계정만 허용한다.

### 상세 플로우

1. 사용자가 회원가입 폼 제출
2. `POST /api/v1/auth/signup`
3. 서버가:
   - 이메일 중복 확인
   - 비밀번호 해시 생성
   - `User(emailVerifiedAt = null)` 생성
   - `EmailVerificationToken` 생성
   - 인증 메일 발송
4. 응답:
   - `201 Created`
   - 프론트에는 `인증 메일을 확인해 주세요.` 표시
5. 사용자가 메일의 인증 링크 클릭
6. `GET /api/v1/auth/verify-email?token=...`
7. 서버가:
   - 토큰 해시 검증
   - 만료/사용 여부 확인
   - `user.emailVerifiedAt = now()`
   - 토큰 `usedAt` 기록
8. 인증 성공 후 로그인 화면 또는 메인으로 이동

### 로그인 정책

- 일반 이메일 로그인:
  - `emailVerifiedAt`가 없으면 로그인 거부
  - 메시지: `이메일 인증 후 로그인해 주세요.`
- Google 로그인:
  - Google 쪽에서 이메일 검증이 끝난 계정만 허용
  - 최초 로그인 시 `emailVerifiedAt`를 채워도 됨

## Cloudflare DNS 설정

실제 값은 Resend가 도메인 추가 후 보여주는 값을 그대로 넣는다.

### 1. 발송용 도메인 인증

Cloudflare DNS에 다음 성격의 레코드를 넣는다.

- SPF용 TXT
- DKIM용 CNAME 여러 개
- 필요 시 DMARC용 TXT

예시 형태:

```txt
Type: TXT
Name: @
Value: (Resend가 안내하는 SPF 값)
```

```txt
Type: CNAME
Name: resend._domainkey
Value: (Resend가 안내하는 DKIM 값)
```

```txt
Type: TXT
Name: _dmarc
Value: v=DMARC1; p=none; rua=mailto:dmarc@songsai.org
```

주의:

- SPF, DKIM 값은 예시가 아니라 Resend 대시보드에서 나온 값을 그대로 넣는다.
- Cloudflare Proxy는 메일 관련 레코드에서 켜지지 않도록 DNS only 상태를 유지한다.

### 2. 수신용 도메인 또는 서브도메인

문의 메일은 루트 도메인보다 서브도메인 분리를 권장한다.

권장 예:

- 발송: `songsai.org`
- 수신: `mail.songsai.org` 또는 `support.songsai.org`

수신 구성 시 필요한 레코드:

- MX
- TXT
- 필요 시 추가 검증 레코드

이 역시 Resend가 inbound 설정 시 보여주는 값을 그대로 Cloudflare에 넣는다.

## Resend 설정 순서

### 발송 설정

1. Resend 계정 생성
2. Domain 추가
   - `songsai.org` 또는 `auth.songsai.org`
3. Domain verification 값 확인
4. Cloudflare DNS에 SPF/DKIM 추가
5. Resend에서 도메인 인증 완료 확인
6. 발신 주소 생성
   - `auth@songsai.org`
   - `no-reply@songsai.org`

### 수신 설정

1. Resend에서 Receiving/Inbounds 활성화
2. 수신 도메인 또는 서브도메인 추가
   - 예: `support.songsai.org`
3. Cloudflare DNS에 MX/TXT 추가
4. Resend webhook endpoint 등록
   - 예: `POST /api/v1/inbound-email/webhook`
5. 테스트 메일 발송 후 webhook 수신 확인

### 운영 권장

- 발송용과 수신용을 분리한다.
- 인증 메일은 `auth@songsai.org`
- 문의 메일은 `support@songsai.org`
- 자동 알림은 `no-reply@songsai.org`

## songsai-api 환경변수

예시:

```env
RESEND_API_KEY=
RESEND_FROM_EMAIL=auth@songsai.org
RESEND_REPLY_TO=support@songsai.org
EMAIL_VERIFY_BASE_URL=https://songsai.org
```


### 이메일 인증 토큰

```prisma
model EmailVerificationToken {
  id         String   @id @default(cuid())
  userId     String
  tokenHash  String   @unique
  expiresAt  DateTime
  usedAt     DateTime?
  createdAt  DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, expiresAt])
}
```

### 발송 이력

### 수신 메일

```prisma
model InboundEmail {
  id             String   @id @default(cuid())
  messageId      String?  @unique
  fromEmail      String
  fromName       String?
  toEmail        String
  subject        String?
  textBody       String?
  htmlBody       String?

  @@index([status, receivedAt])

```

### 수신 첨부파일

## songsai-api 코드 구조 제안

### 서버 모듈

추가 권장 파일:

- `src/server/email/provider.ts`
  - Resend 발송 래퍼
- `src/server/email/templates.ts`
  - 인증 메일 제목/본문 템플릿
- `src/server/email/tokens.ts`
  - 토큰 발급/해시/만료 검증
- `src/server/email/inbound.ts`
  - webhook payload 정규화 및 DB 저장

### 인증 관련 API

- `POST /api/v1/auth/signup`
  - 기존 회원가입 + 인증 메일 발송
- `POST /api/v1/auth/resend-verification`
  - 미인증 계정 재발송
- `GET /api/v1/auth/verify-email`
  - 토큰 검증
- `POST /api/v1/auth/login`
  - 미인증 계정 로그인 차단

### 메일 webhook API

- `POST /api/v1/inbound-email/webhook`
  - Resend inbound webhook 수신
- `POST /api/v1/email/webhook`
  - 발송 결과 webhook 저장이 필요하면 별도 사용

## 인증 메일 발송 로직

### 회원가입 시

1. `signupSchema` 검증
2. 이메일 중복 확인
3. `User(emailVerifiedAt = null)` 생성
4. 랜덤 토큰 생성
5. DB에는 `tokenHash` 저장
6. 사용자에게는 raw token이 들어간 링크 발송
7. `OutboundEmailLog` 저장

### 인증 링크

예시:

```txt
https://songsai.org/verify-email?token=RAW_TOKEN
```

백엔드 처리:

- raw token 해시
- `EmailVerificationToken.tokenHash` 비교
- `expiresAt > now()`
- `usedAt == null`
- 성공 시:
  - `user.emailVerifiedAt = now()`
  - `token.usedAt = now()`

## 로그인 차단 정책

일반 로그인 시:

- `emailVerifiedAt == null`
  - `403`
  - 메시지: `이메일 인증 후 로그인해 주세요.`

프론트 표시:

- 로그인 실패 팝업/에러 영역에 위 문구 표시
- 함께 `인증 메일 다시 보내기` 버튼 제공 가능

## webhook 엔드포인트 설계

### 경로

- `POST /api/v1/inbound-email/webhook`

### 검증

- Resend webhook secret 검증
- 서명 헤더 또는 secret 비교
- 실패 시 `401`

### 저장 흐름

1. webhook payload 수신
2. `messageId`, `from`, `to`, `subject`, `text`, `html`, `attachments` 추출
3. `InboundEmail` upsert 또는 create
4. 첨부파일이 있으면 로컬 저장 후 `InboundEmailAttachment` 생성
5. `status = NEW`

### 응답

```json
{ "ok": true }
```

## 구현 순서 제안

### 1차

- Prisma 스키마 추가
- Resend 발송 연동
- 회원가입 인증 메일 발송
- 이메일 인증 완료 처리
- 미인증 로그인 차단

### 2차

- 인증 메일 재발송
- 발송 로그 저장
- 관리자용 인증 상태 조회

### 3차

- Resend inbound webhook
- 수신 메일 DB 저장
- `/admin/inbox` 목록/상세 화면

### 4차

- 첨부파일 저장
- 답변 상태 관리
- 운영용 검색/필터/통계

## 주의사항

- 토큰 raw 값은 DB에 저장하지 않고 해시만 저장한다.
- 인증 링크는 만료시간을 둔다.
- 이미 인증된 사용자의 토큰 재사용은 무시하거나 안내 페이지로 보낸다.
- 메일 발송 실패 시 회원가입 자체를 롤백할지, 미인증 계정만 남길지 정책을 정해야 한다.

권장 정책:

- 회원은 생성하되 `emailVerifiedAt = null`
- 메일 발송 실패 시 사용자에게 재발송 유도

## 이번 설계의 결론

- `Cloudflare`는 DNS 관리
- `Resend`는 발송/수신/webhook
- `songsai-api`는 인증 토큰, 발송 로직, inbound 저장, 관리자 화면 API 담당
- 일반 회원가입은 이메일 인증 완료 후 실사용 가능
- 수신 메일은 DB에 저장해서 관리자 화면에서 조회
